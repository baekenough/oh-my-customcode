# R020: Completion Verification Rules

Before declaring any task `[Done]`, verify actual completion against task-type-specific criteria. Claiming completion without verification is a violation.

## Overview

False completion declarations erode trust and cause downstream failures. "Ran the command" does not equal "succeeded." Each task type has explicit required verifications that must be confirmed before `[Done]` is declared.

## Priority

**MUST** — Non-negotiable. Declaring `[Done]` without verification evidence is a violation requiring immediate correction.

## Key Requirements

**Task-type completion matrix:**

| Task Type | Required Verification |
|-----------|----------------------|
| Release | All issues closed, version bumped, PR merged, GitHub Release created, external automation verified |
| Implementation | Code compiles, tests pass, no TODO markers left |
| Documentation | Links valid, counts accurate, cross-references updated |
| Git Operations | Operation succeeded (exit code checked), working tree clean |
| Code Review | All findings addressed or explicitly deferred |
| Agent/Skill Creation | Frontmatter valid, referenced skills exist, routing updated |

**Completion contract format (complex tasks):**
```
[Contract] Task: {name}
├── Criterion 1: {specific, verifiable condition}
└── Criterion N: {specific, verifiable condition}
```

**Done declaration format:**
```
[Done] Task: {name}
├── ✓ Criterion 1: {evidence}
└── ✓ Criterion N: {evidence}
```

**Common false completion patterns to avoid:**

| Pattern | Reality | Fix |
|---------|---------|-----|
| "Command executed" | Exit code not checked | Check `$?` or tool output |
| "File created" | Content not verified | Read file back, verify content |
| "Tests pass" | Only ran subset | Run full test suite |
| "Auto-publish ran" | External workflow not confirmed | Check `gh run list` for triggered jobs |
| "Subagent reported pre-existing" | Not verified against baseline | Compare develop branch directly |

## Subagent Self-Report Verification

Subagents may claim failures were "pre-existing" to avoid blame. The orchestrator must verify such claims before accepting them.

| Subagent Claim | Verification Method |
|----------------|---------------------|
| "This bug pre-existed the change" | `git diff develop` on the relevant file |
| "That test was already failing" | Run test suite on develop branch |
| "External workflow already triggered" | `gh run list --workflow=<name>` |
| "Issue was already closed" | Check GitHub issue timeline directly |

**Rule:** "Pre-existing" claims are hypotheses, not facts. Verify before accepting.

## Autonomous Mode Entry Checklist

When entering autonomous mode (user grants extended execution without per-step confirmation), perform this inventory BEFORE first action:

1. **Workflow inventory**: `ls .github/workflows/` — identify auto-publish, auto-tag, release, docs-sync, CI workflows
2. **Recent runs**: `gh run list --limit 10` — check success/failure patterns of automated workflows
3. **External publish targets**: Check if npm/PyPI/Docker Hub/GitHub Releases are auto-triggered on merge
4. **Manual intervention points**: Identify which steps require human approval vs. fully automated
5. **Cross-reference with task**: Which workflows will the planned work trigger?

Record findings in session context. Failure to inventory automation is a R020 violation (unknown external state = unverifiable completion).

## Enforcement

Prompt-based per [[r021]]. Integrates with [[r003]] `[Done]` format and [[r017]] structural verification.
