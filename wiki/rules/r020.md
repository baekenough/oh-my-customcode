# R020: Completion Verification Rules

Before declaring any task `[Done]`, verify actual completion against task-type-specific criteria. Claiming completion without verification is a violation.

## Overview

False completion declarations erode trust and cause downstream failures. "Ran the command" does not equal "succeeded." Each task type has explicit required verifications that must be confirmed before `[Done]` is declared.

## Priority

**MUST** — Non-negotiable. Declaring `[Done]` without verification evidence is a violation requiring immediate correction.

## Key Requirements

**Task-type completion matrix:**

| Task Type | Required Verification |
|-----------|----------------------|
| Release | All issues closed, version bumped, PR merged, GitHub Release created, external automation verified |
| Implementation | Code compiles, tests pass, no TODO markers left |
| Documentation | Links valid, counts accurate, cross-references updated |
| Git Operations | Operation succeeded (exit code checked), working tree clean |
| Code Review | All findings addressed or explicitly deferred |
| Agent/Skill Creation | Frontmatter valid, referenced skills exist, routing updated |
| UI/Frontend | Browser render verified (dev server running + page loaded), no console errors, visual output matches intent; CSS/style changes require before/after visual diff; type-check passing alone is NOT sufficient |

**Completion contract format (complex tasks):**
```
[Contract] Task: {name}
├── Criterion 1: {specific, verifiable condition}
└── Criterion N: {specific, verifiable condition}
```

**Done declaration format:**
```
[Done] Task: {name}
├── ✓ Criterion 1: {evidence}
└── ✓ Criterion N: {evidence}
```

**Common false completion patterns to avoid:**

| Pattern | Reality | Fix |
|---------|---------|-----|
| "Command executed" | Exit code not checked | Check `$?` or tool output |
| "File created" | Content not verified | Read file back, verify content |
| "Tests pass" | Only ran subset | Run full test suite |
| "Auto-publish ran" | External workflow not confirmed | Check `gh run list` for triggered jobs |
| "Subagent reported pre-existing" | Not verified against baseline | Compare develop branch directly |
| "UI changes done" / "CSS updated" | type-check passes but browser render not verified | Start dev server, open browser, confirm visual output |

## Subagent Self-Report Verification

Subagents may claim failures were "pre-existing" to avoid blame. The orchestrator must verify such claims before accepting them.

| Subagent Claim | Verification Method |
|----------------|---------------------|
| "This bug pre-existed the change" | `git diff develop` on the relevant file |
| "That test was already failing" | Run test suite on develop branch |
| "External workflow already triggered" | `gh run list --workflow=<name>` |
| "Issue was already closed" | Check GitHub issue timeline directly |

**Rule:** "Pre-existing" claims are hypotheses, not facts. Verify before accepting.

## Autonomous Mode Entry Checklist

When entering autonomous mode (user grants extended execution without per-step confirmation), perform this inventory BEFORE first action:

1. **Workflow inventory**: `ls .github/workflows/` — identify auto-publish, auto-tag, release, docs-sync, CI workflows
2. **Recent runs**: `gh run list --limit 10` — check success/failure patterns of automated workflows
3. **External publish targets**: Check if npm/PyPI/Docker Hub/GitHub Releases are auto-triggered on merge
4. **Manual intervention points**: Identify which steps require human approval vs. fully automated
5. **Cross-reference with task**: Which workflows will the planned work trigger?

Record findings in session context. Failure to inventory automation is a R020 violation (unknown external state = unverifiable completion).

## Optional: Quantitative Evidence (advisory, v0.114.0, #1034)

For complex agent invocations or multi-step workflows, 4-metric evidence from the `agent-eval-framework` skill can be attached to `[Done]` declarations as supplementary evidence (NOT a binary gate). Metrics: `correctness`, `step_ratio`, `tool_call_ratio`, `latency_ratio`. See source rule for full table and when-to-apply guidance.

## Interrupt ≠ Prior-Request Cancellation (#1341 ①)

Read-Before-Characterize의 인터럽트-의도 확장. 인터럽트 직후 첫 메시지가 모호하면 직전 요청을 "취소"로 단정하지 않는다.

**적용 범위: 비파괴적 작업에 한함.** 진행 중 작업이 파괴적·비가역(R001 — `git reset --hard`, `git clean -fd`, `rm`, 터널/DNS/k8s 삭제 등)이면 이 섹션은 적용되지 않는다 — Safety Carve-Out이 우선한다.

| 상황 | 금지 | 필수 |
|------|------|------|
| 인터럽트 직후 모호한 첫 메시지 (비파괴적 작업) | 직전 요청 취소로 단정하고 새 작업 시작 | 멀티라인/연속 입력 가능성 고려 — 의도 1회 확인 |
| "인터럽트 = 취소" 자동 가정 | 금지 — 멀티라인 입력·교정·추가 요청일 수 있음 | 모호하면 `[Confirm] ...` 확인 후 진행 |

**모호성 판정 신호:** (a) 직전 요청이 미완료 상태에서 인터럽트 발생, (b) 첫 메시지가 직전 요청과 무관하지만 새 독립 명령으로도 단정 불가. 둘 다 해당하지 않으면 [[r003]] Clear 경로(즉시 실행)를 우선한다 — 과잉 확인 방지.

**Interrupt Priority Re-Ordering과의 구분:** Priority Re-Ordering은 인터럽트가 명확한 새 작업 또는 룰 위반 지적을 동반할 때 적용한다. 첫 메시지 자체가 모호한 경우는 본 섹션이 우선한다.

### Safety Carve-Out (fail-closed, R001 우선)

진행 중 작업이 파괴적·비가역인 경우, 인터럽트 의도가 모호하더라도 **먼저 즉시 halt** 후 의도를 확인한다. "진행" 허용은 비파괴적 작업에만 성립하며, 파괴적 작업의 계속을 의미하지 않는다.

| 상황 | 행동 |
|------|------|
| 파괴적 작업 중 인터럽트 (의도 모호) | 즉시 halt → `[Confirm] ...` 의도 확인 → 명시적 재승인 후 재개 |
| 비파괴적 작업 중 인터럽트 (의도 모호) | `[Confirm] ...` 의도 확인 (halt 불필요) |
| 파괴적 작업 중 인터럽트 (명확한 새 지시) | halt → 새 지시 처리 |
| 비파괴적 작업 중 인터럽트 (명확한 새 지시) | [[r003]] Clear 경로 즉시 처리 |

**R001 파괴적 작업 목록 (안전 carve-out 적용):** `git reset --hard`, `git clean -fd`, `git push --force` (공유 브랜치), `git branch -D` (미머지 커밋 포함), 광범위 `rm`/삭제, 터널/DNS/k8s 리소스 삭제, prod 실행 변경.

Cross-reference: [[r003]] "Interrupt (ambiguous first message)" 행, [[r001]] 파괴적 작업 목록.

## Interrupt Priority Re-Ordering (v0.147.0)

사용자 인터럽트 / 새 요청 / 룰 위반 지적 동시 수신 시, 사과 한 줄 후 즉시 작업 통합 plan 재정렬. "사과만 짧게 + 기존 흐름 유지" 패턴 금지.

| 시나리오 | Required 행동 |
|---------|--------------|
| 새 작업 + 룰 위반 지적 동시 | (1) 룰 위반 즉시 수정 (2) 새 작업 통합 plan 제시 (3) 사용자 확인 후 진행 |
| 인터럽트 후 기존 작업 진행 | 인터럽트 내용 통합 또는 명시적 deferral 후 진행 |
| "사과만 짧게" | 부족 — plan 재정렬 후속 필수 |

Reference issues: #1188 item #8.

## Diagnostic Hypothesis Verification (#1217 item #4)

진단 단계에서 채택한 가설로 워크플로우/인프라/설정을 **영구 변경하기 전**, 실제 증거로 가설을 검증해야 한다. 그럴듯한 가설을 검증 없이 영구 변경에 적용하면 잘못된 추정이 영구 부채로 남는다.

| 상황 | 금지 | 필수 |
|------|------|------|
| 에러 원인 추정 | 첫 가설로 워크플로우/설정 영구 수정 | 가설을 좁은 범위에서 검증 후 변경 |
| CI/publish 실패 | 추정 기반 우회 커밋 머지 | 에러 메시지/로그로 실제 원인 확정 |
| 권한/토큰 오류 | 플래그/옵션 변경으로 우회 시도 | 권한 범위·토큰 종류 직접 확인 |

**Common Violation (#1217 item #4):** npm publish E403을 `--provenance` attestation 충돌로 오진단 → release workflow에서 `--provenance` 제거 커밋 머지 → 2차 시도 동일 실패 → 실제 원인은 NPM_TOKEN 권한(Automation token 필요). 잘못된 추정으로 릴리즈 워크플로우를 영구 변경.

**영구 변경 전 self-check:**
1. 가설을 뒷받침하는 직접 증거(로그/에러 코드/문서)가 있는가?
2. 비파괴적 방법으로 가설을 검증할 수 있는가?
3. 변경이 되돌리기 쉬운가? (영구 워크플로우 변경 vs 일회성 시도)

하나라도 NO면 검증을 먼저 수행한다. 근본 원인 진단은 `superpowers:systematic-debugging` 참조.

### Config-Schema-Before-Edit (v0.177.0, #1327 찐빠 #2)

변경이 **오버라이드/우선순위/상속 체인에 참여하는 필드**(예: provider + base_url, multi-key fallback, layered defaults)를 건드릴 때 적용된다. 플래그 하나 토글하거나 타임아웃 값만 바꾸는 독립 필드 편집에는 전체 스키마 읽기가 불필요하다.

대상 편집에 체인 참여 필드가 포함된 경우, **전체 config 스키마와 오버라이드-우선순위 체인**을 먼저 읽은 뒤 계획한다.

| 금지 | 필수 |
|------|------|
| config 오버라이드 체인을 읽지 않고 provider/endpoint 전환을 N개 명령으로 계획 | 전체 config 스키마 읽기(어느 필드가 이기는지, 기본값, 상속) → 전환이 건드리는 모든 필드 열거(base_url 포함) → 그 후 계획 |

**Common Violation (#1327):** DeepSeek provider 전환 시 auth + provider + default 3-command 편집을 계획했으나 `base_url`을 누락 — 기존 provider(openrouter.ai)를 계속 가리키게 될 뻔했다. config의 base_url 오버라이드-우선순위를 계획 전에 읽지 않은 것이 원인.

Read-Before-Characterize의 형제 규칙이다(Read-Before-Characterize는 진단 — 읽기 전 레이블 금지; Config-Schema-Before-Edit는 편집 계획 완전성 — 편집 전 모든 의존 필드 열거). Cross-ref: [[r023]] (verification ladder — config 완전성은 Tier-1 결정론적 사전 검사).

### Proxy Signal vs Canonical Ground-Truth (#1336 ①②)

When diagnosing pipeline or data state, verify the **canonical store** (authoritative DB / system-of-record) BEFORE characterizing state from a secondary proxy (a `.txt` file artifact, filesystem mtime) or a single ingestion path. Two failure modes share this meta-pattern:

| Anti-pattern | Required |
|--------------|----------|
| Characterize pipeline health from a filesystem proxy (`.txt` presence/mtime) | Query the canonical store (DB count/recency) first |
| Generalize one ingestion path's failure (one empty XML / one 401) to "whole pipeline blocked" | Check the final landing store's count across ALL paths before concluding blockage |

A single path's failure does NOT prove the whole multi-path pipeline is down. Confirm the system-of-record before alarming or dispatching reprocessing.

**Common Violations (#1336):** (①) Transcription was alarmed as "stopped" because `.txt` files looked stale — the canonical DB had transcripts current to 06-09 21:30 (the whisper collector emits only to the DB, not to `.txt`). (②) SMS was over-diagnosed as "fully blocked" from one empty OneDrive XML path + a single 401, while the DB held 17 SMS rows ingested via the app path.

### Read-Before-Characterize (#1266 ④)

진단 대상(로그, 출력, 데이터)을 **충분히 읽기 전에** 에러 클래스나 원인을 단정하지 않는다.

| 금지 | 필수 |
|------|------|
| 로그/출력을 읽기 전 "error loop"·"무한 루프"로 특성화 | 대표 샘플을 먼저 읽고 INFO/WARN/ERROR 분포 확인 후 특성화 |
| 첫 namespace/scope만 보고 전체 단정 | 관련 scope 확인 후 결론 |
| 정렬 기준 미검증 시계열 단정 (`ls\|tail`로 "최신") | 시계열 판단은 mtime/timestamp 정렬(`ls -t`/`find -newermt`) 명시 후 결론 — 파일명순 ≠ 시간순 (#1417) |

Origin: #1266 ④.

## CI Publish-Step Error vs Published-Artifact Ground Truth (v0.178.0, #1332)

A CI publish/deploy step that LOGS an error has NOT necessarily failed. The step may recover via a fallback (`|| npm view ...`), or the error may be in a non-fatal sub-step (provenance attestation, eventual-consistency probe). Before declaring a publish/release run failed — and ESPECIALLY before re-running, rolling back, or permanently changing the workflow — verify the PUBLISHED ARTIFACT directly:

| Publish target | Ground-truth check |
|----------------|--------------------|
| npm | `npm view <pkg> version` == expected |
| GitHub Release | `gh release view <tag>` exists, not draft |
| Docker registry | image tag/manifest exists |
| Run outcome | `gh run view <id> --json jobs` job conclusions — NOT a single step's log line |

This is the publish-domain extension of Read-Before-Characterize ("actual outcome ≠ attempt"). Re-running a publish that actually succeeded risks duplicate-publish errors; permanently changing a workflow on a misdiagnosis compounds the error (cf. #1217 — npm E403 misdiagnosed as `--provenance` conflict → wrong workflow change → repeated failure).

**Common Violation (#1332):** `npm publish --provenance` emitted a Sigstore TLOG 409 error; a subagent read the error in the step log and declared the run "failed," recommending a re-run. The publish step's `|| npm view` fallback had already recovered — the package was published and `gh release view` confirmed the release fully succeeded. Deterministic artifact verification was not performed before declaring failure.

## Test-Skip Is Not Completion (#1217 item #5)

테스트 실패를 `describe.skip`/`xfail`/`it.skip` + coverage threshold 하향으로 회피하는 것은 완료가 아니다. 근본 원인 분석 없이 그린 빌드를 만드는 회피는 기술 부채를 다음 마이너로 이월시킨다.

| 금지 | 필수 |
|------|------|
| 실패 테스트 skip + threshold 하향으로 그린 빌드 | 근본 원인 분석 후 수정, 불가 시 명시적 deferral + 이슈 등록 |
| "다음 버전 TODO" 주석만 남기고 머지 | skip 사유·복구 조건·추적 이슈를 함께 기록 |

## State-Change Claim → Live System Verification (#1335 ①)

Before closing or marking-done an issue/task that CLAIMS an infrastructure or resource **state change** (a service stopped, a resource torn down, a deployment removed, a process killed), verify the ACTUAL live system state — not just that the change command was issued. "Issued the teardown" ≠ "the resource is down."

| Claimed state change | Live ground-truth check |
|----------------------|-------------------------|
| LaunchAgent/service stopped | `launchctl list` / `systemctl status` shows it absent/inactive |
| k8s resource torn down | `kubectl get <resource>` returns NotFound |
| Container removed | `docker ps -a` does not list it |
| Process killed | `pgrep`/`ps` returns empty |

This is the infra/state extension of "actual outcome ≠ attempt." Closing on the command-issued assumption leaves orphaned running resources.

**Common Violation (#1335 ①):** Issue #101 (secretary teardown) was closed as "대체 완료·teardown 보류", but the secretary LaunchAgents (onedrive-bridge / calendar-worker / minikube-mount) were still running on the host. The user caught it — they were not torn down.

## Binary/Rendered-Artifact Completeness (text-grep ≠ complete) (#1384)

완료/완결성을 주장하는 작업(redaction, 식별자 제거, 콘텐츠 정리, 시크릿 스크럽, 데이터 마이그레이션)에서 텍스트 grep 통과는 완결을 보장하지 않는다. 렌더된 이미지/바이너리 산출물(PNG/PDF/렌더 다이어그램/임베디드 메타데이터/EXIF)에 동일 대상이 시각적·바이너리적으로 잔존할 수 있다.

"완전 제거됨/완료" 선언 전, 텍스트뿐 아니라 바이너리/이미지/렌더 산출물 완결성까지 검증해야 한다. Task-Type Completion Matrix의 UI/Frontend "browser render verified" 행의 redaction/binary 확장이다.

| Anti-pattern | Required |
|--------------|----------|
| 텍스트 grep 통과 후 "완전 제거됨/정리됨" 선언 | 렌더 이미지/바이너리/임베디드 메타데이터 시각·내용 스캔까지 통과한 뒤 선언 |
| redaction 범위를 grep 가능 텍스트로만 한정 → 잔여를 순차 발견하며 force-push 반복 | 사전 전수 점검(텍스트 + 부분문자열 변형 + 바이너리/이미지 + 참조/고아 분석) 후 단일 패스 rewrite |

**Common Violation (#1384):** 텍스트 + git 히스토리 force-push 후 "원격 완전 정리됨" 선언 → 직후 렌더된 다이어그램 PNG 3종에 redaction 대상 식별자가 시각적으로 잔존 + 텍스트 잔여 토큰 1건 발견 → 추가 force-push 2회 필요. redaction 범위를 grep 가능한 텍스트로만 잡고 렌더 이미지를 완결 선언 전에 점검하지 않은 것이 원인.

Cross-reference: [[r001]] (보안 완결성 — 시크릿/식별자 잔존 차단), [[r005]] (단일 패스 효율 — 사전 전수 점검이 반복 force-push를 방지).
