# R010: Orchestrator Coordination Rules

The main conversation is the sole orchestrator and MUST NEVER directly write, edit, or create files — all file modifications MUST be delegated to appropriate subagents.

## Overview

The orchestrator's role is routing and coordination only. File operations, git commands, and all write actions must go through specialist subagents. This ensures separation of concerns, maintains audit trails, and prevents accidental bulk changes from the orchestrator context.

## Priority

**MUST** — Critical. Direct file writes from the orchestrator are violations. Advisory PostToolUse hook monitors compliance. Candidate for hard-block promotion per [[r021]].

## Key Requirements

**Delegation table (key entries):**

| Task Type | Required Agent |
|-----------|---------------|
| Create agent/skill/guide | `mgr-creator` |
| Git operations | `mgr-gitnerd` |
| Memory operations | `sys-memory-keeper` |
| Python/FastAPI | `lang-python-expert` / `be-fastapi-expert` |
| Go code | `lang-golang-expert` |
| TypeScript/Next.js | `lang-typescript-expert` / `fe-vercel-agent` |
| Docker/Infra | `infra-docker-expert` |
| Database schema | `db-supabase-expert` |
| Unmatched specialized task | `mgr-creator` → dynamic agent creation |

**Protected paths (mgr-creator required):**
- `.claude/agents/*.md`
- `.claude/skills/*/SKILL.md`
- `guides/*/` (new directories)

**Orchestrator allowed operations:** Read, Glob, Grep only (read-only).

**Dynamic agent creation:** When no matching agent exists → delegate to `mgr-creator` with domain keywords, file patterns, and required capabilities. mgr-creator auto-discovers relevant skills/guides and creates the agent.

**Agent Teams exception:** Agent Teams members are peers and CAN spawn sub-agents to execute complex workflows.

**Sub-agent nesting (v2.1.172+, platform capability):** The CC platform now allows sub-agents to spawn their own sub-agents up to 5 levels deep. oh-my-customcode RETAINS the sole-orchestrator design (subagents do not spawn subagents via the Agent tool) as a deliberate project architecture choice — for predictable R009 parallelism and R018 coordination — NOT a platform limitation. The only sanctioned nesting path is the Agent Teams exception above.

## Agent Capability Pre-Check (v0.147.0)

Before delegating a task to a subagent, MUST verify the target agent's tool capabilities against the task requirements. Failure to pre-check causes round-trip waste (delegation → failure → re-delegation).

| Task involves | Verify in target agent frontmatter |
|--------------|-----------------------------------|
| `gh` / shell commands | `tools:` includes Bash AND `disallowedTools:` excludes Bash |
| `Read` external files | `tools:` includes Read |
| `Write` files | `tools:` includes Write |
| MCP server calls | `mcpServers:` includes the required server |

**Known limitation:** `arch-documenter` has `disallowedTools: [Bash]` — cannot run `gh` or shell scripts. Pre-collect data via orchestrator and pass as content, or use `general-purpose` for the Bash-needing portion.

Reference issues: #1202 item #2, `feedback_arch_documenter_no_bash.md`.

## Sensitive Path Handling

> **Status (CC v2.1.121+):** DEPRECATED. Direct Write/Edit/Bash on `.claude/`, `.git/`, `.vscode/` works without prompts under `mode: "bypassPermissions"`. The `/tmp/*.sh` bypass pattern previously required is no longer necessary. See #1101.

The legacy `/tmp/*.sh` script bypass pattern applied to CC < v2.1.121 only. For current CC versions, `mode: "bypassPermissions"` on every Agent tool call (see Universal bypassPermissions) is sufficient.

**Background-agent project-settings isolation (v2.1.172+):** Fixed background agents potentially reading another directory's project settings (`.mcp.json` approvals, trust) when dispatched onto a pre-warmed worker. A `/bg`-dispatched agent now correctly reads its own project's settings. Strengthens background-agent isolation for unattended `/bg` flows.

**Background-agent provider env isolation (v2.1.174+):** Fixed background sessions inheriting another session's `ANTHROPIC_*` provider env (gateway URL, custom headers, `/model` aliases) from the shell that started the background daemon. A `/bg`-dispatched agent no longer picks up a foreign session's provider configuration. Also fixed pre-warmed background workers failing with "Could not resolve authentication method" when claimed after sitting idle. `mode: "bypassPermissions"` on every Agent tool call remains required regardless. This further strengthens background-agent isolation (cf. v2.1.172 project-settings isolation above).

## Pre-Delegation Privileged-Scope Boundary (proactive)

> Origin: #1368 #5 — an infra subagent was delegated a prod-touching task with NO explicit approval boundary in the delegation prompt; it freely ran prod DB queries, file deletes, and SMS reads, tripping the safety classifier 3+ times.

The **Subagent Scope-Creep STOP Protocol is REACTIVE** — it halts an agent after classifier trips. This is its PROACTIVE complement: prevent the trips by stating the boundary before the subagent runs.

When delegating ANY task that touches prod or privileged resources (prod DB, infra deletion, credential stores, external messaging/SMS, shared-namespace secrets), the orchestrator MUST state — **explicitly in the delegation prompt** — the approved actions, forbidden actions, and authorization scope.

| Anti-pattern | Required |
|--------------|----------|
| Delegate a prod/privileged-touching task with no scope or forbidden-line in the prompt | State in the prompt: approved action(s), explicit forbidden actions (e.g. "do NOT delete files, do NOT query prod DB, do NOT read SMS/messages"), and authorization scope |

Cross-reference: Subagent Scope-Creep STOP Protocol (reactive halt after trips), [[r001]] (credential/privileged-scope guardrails).

## CC Platform Updates

**Auto mode subagent safety classifier (v2.1.178+):** Auto mode now evaluates subagent spawns with the safety classifier BEFORE launch. This is a platform-level defense-in-depth complement to the advisory Pre-Delegation Privileged-Scope Boundary (above) — the platform itself can block a subagent spawn before it runs, independently of the orchestrator's pre-delegation checks. The advisory model-level Pre-Delegation Boundary remains required (R021 advisory-first), but the platform now adds a reactive gate as well.

**`claude agents` 401 under custom API gateway (v2.1.178+):** Fixed `claude agents` workers returning a 401 `Invalid bearer token` when a custom API gateway is configured via `ANTHROPIC_BASE_URL` + `ANTHROPIC_AUTH_TOKEN`. Also fixed `/bg`-dispatched background agents getting stuck showing "Working" forever in that configuration. Both were caused by the worker endpoint not honoring the custom auth token; v2.1.178 corrects this.

**Remote session "still running" display fix (v2.1.179+):** Fixed background tasks in remote sessions appearing stuck as "still running" between turns. Background agents that had completed a turn were incorrectly shown as still active; v2.1.179 resolves the display freeze. `mode: "bypassPermissions"` on every Agent tool call remains required regardless.

**Custom gateway prompt caching fix (v2.1.181+):** Fixed per-request attestation tokens rotating every turn under custom `ANTHROPIC_BASE_URL` (including Foundry), which prevented prompt caching reads from hitting. This further strengthens background-agent provider isolation (cf. v2.1.172 project-settings isolation, v2.1.174 ANTHROPIC_* env isolation, v2.1.178 401 bearer-token fix). If using a custom API gateway, upgrade to v2.1.181+ for stable prompt caching. `mode: "bypassPermissions"` on every Agent tool call remains required regardless.

**`!` bash auto-respond + background interrupt fix (v2.1.186+):** `!` bash commands now trigger Claude to respond to the output automatically (set `"respondToBashCommands": false` to keep the prior context-only behavior). Also fixed Esc/Ctrl+C not responding while background agents run after the main turn ends. `mode: "bypassPermissions"` on every Agent tool call remains required.

**Subagent depth tracking + background-job fixes (v2.1.187+):** Fixed subagent depth tracking — resumed subagents restore their original spawn depth, and forked subagents count toward the depth cap. Also fixed background jobs stuck in "working" indefinitely when an agent ended a turn without structured output, and leaked agent worktree registrations are now auto-cleaned. Reinforces the sole-orchestrator design's depth accounting (cf. v2.1.172 5-level nesting note). `mode: "bypassPermissions"` remains required.

**Background agent stop permanence (v2.1.191+):** Stopping a background agent from the tasks panel is now PERMANENT — a stopped agent no longer resurrects after being stopped. Strengthens background-agent lifecycle control. `mode: "bypassPermissions"` on every Agent tool call remains required.

## Enforcement
