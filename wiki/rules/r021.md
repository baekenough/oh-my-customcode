# R021: Enforcement Policy

oh-my-customcode uses an advisory-first enforcement model. Most rules are enforced through prompt engineering rather than hard-blocking hooks, preserving agent flexibility while maintaining behavioral standards.

## Overview

Hard-blocking hooks are reserved for critical safety boundaries. The majority of MUST rules are enforced through PostToolUse advisory hooks, PostCompact re-injection, and CLAUDE.md prompt guidance. This prevents agents from getting trapped in unrecoverable states while still maintaining standards.

## Priority

**MUST** — The enforcement policy itself defines how all other rules are enforced.

## Key Requirements

**Enforcement tiers:**

| Tier | Mechanism | Rules | Behavior |
|------|-----------|-------|----------|
| Hard Block | PreToolUse hook, exit 2 | stage-blocker, dev-server tmux, rule-deletion-guard | Prevents tool execution |
| Soft Block | Stop hook prompt | [[r011]] session-end saves | Auto-performs then approves |
| Advisory | PostToolUse hooks | [[r007]], [[r008]], [[r009]], [[r010]], [[r018]] | Warns via stderr, never blocks |
| Prompt-based | CLAUDE.md + rules/ + PostCompact | All MUST rules | Behavioral guidance in context |

**Why advisory-first:**
1. Hard blocks can trap agents in unrecoverable states
2. Missing dependencies (jq, etc.) don't break the session
3. External skills and internal rules can coexist
4. PostCompact re-injection maintains rules after context compaction

**Hard-block promotion conditions:**

| Rule | Condition for Promotion |
|------|------------------------|
| [[r010]] | Orchestrator-direct-write violations > 3/session |
| [[r007]]/[[r008]] | Identification omission rate > 20% |

**Promoted to hard block:**
- `rule-deletion-guard.sh` (2026-04-08) — prevents accidental bulk deletion of rule files

## Enforcement

This rule IS the enforcement policy. Self-referential: enforced through its own advisory-first model.
