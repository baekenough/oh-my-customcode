# R002: Permission Rules

Defines tool permission tiers and file access boundaries for all agents.

## Overview

Tools are grouped into six tiers from always-allowed read-only tools to conditionally-available MCP tools. File access is scoped to the project. Agents must not attempt operations beyond their tier without explicit approval.

## Priority

**MUST** — Permission violations can cause unintended state changes or security breaches across the system.

## Key Requirements

**Tool tiers:**

| Tier | Tools | Policy |
|------|-------|--------|
| 1: Always | Read, Glob, Grep, ToolSearch | Free use, read-only |
| 2: Default | Write, Edit, NotebookEdit | State changes; notify before modifying important files |
| 3: Context | Agent, Skill, EnterPlanMode, ExitPlanMode, Monitor, TodoWrite, AskUserQuestion | Context-dependent, no user approval |
| 4: Approval | Bash, WebFetch, WebSearch | Request user approval on first use |
| 5: Conditional | Agent Teams tools (TeamCreate, SendMessage, etc.) | Available when Agent Teams enabled |
| 6: MCP | ListMcpResourcesTool, ReadMcpResourceTool, CronCreate, etc. | Available when MCP servers configured |

**File access:**

| Operation | Allowed | Prohibited |
|-----------|---------|-----------|
| Read | All source, configs, docs | — |
| Write | Source code, new files in project | `.env`, `.git/config`, paths outside project |
| Delete | Temp files created by agent | Existing files without request, entire directories |

**Permission request format:**
```
[Permission Request]
Action: {action} | Required: {tool} | Reason: {why} | Risk: Low/Medium/High
Approve?
```

**Deny rule glob patterns (v2.1.166+):** Deny rules in `settings.json` support glob patterns in the tool-name position — `"*"` denies all tools. Allow rules reject non-MCP globs. Unknown tool names in deny rules emit a startup warning.

**Per-parameter `Tool(param:value)` deny syntax (v2.1.178+):** Deny rules now support `Tool(param:value)` syntax to match a tool's input parameters, with `*` as a wildcard (e.g. `Agent(model:opus)` denies Opus subagent spawns). Extends v2.1.166 tool-name globs to per-parameter granularity. Complements `availableModels` ([[r006]]) for model-level access control and `mode: "bypassPermissions"` ([[r010]]) for subagent scope enforcement.

## Enforcement

Prompt-based per [[r021]]. On insufficient permission: do not attempt, notify user, suggest alternative.
