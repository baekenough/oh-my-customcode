# R002: Permission Rules

Defines tool permission tiers and file access boundaries for all agents.

## Overview

Tools are grouped into six tiers from always-allowed read-only tools to conditionally-available MCP tools. File access is scoped to the project. Agents must not attempt operations beyond their tier without explicit approval.

## Priority

**MUST** — Permission violations can cause unintended state changes or security breaches across the system.

## Key Requirements

**Tool tiers:**

| Tier | Tools | Policy |
|------|-------|--------|
| 1: Always | Read, Glob, Grep, ToolSearch | Free use, read-only |
| 2: Default | Write, Edit, NotebookEdit | State changes; notify before modifying important files |
| 3: Context | Agent, Skill, EnterPlanMode, ExitPlanMode, Monitor, TodoWrite, AskUserQuestion | Context-dependent, no user approval |
| 4: Approval | Bash, WebFetch, WebSearch | Request user approval on first use |
| 5: Conditional | Agent Teams tools (TeamCreate, SendMessage, etc.) | Available when Agent Teams enabled |
| 6: MCP | ListMcpResourcesTool, ReadMcpResourceTool, CronCreate, etc. | Available when MCP servers configured |

**File access:**

| Operation | Allowed | Prohibited |
|-----------|---------|-----------|
| Read | All source, configs, docs | — |
| Write | Source code, new files in project | `.env`, `.git/config`, paths outside project |
| Delete | Temp files created by agent | Existing files without request, entire directories |

**Permission request format:**
```
[Permission Request]
Action: {action} | Required: {tool} | Reason: {why} | Risk: Low/Medium/High
Approve?
```

**Deny rule glob patterns (v2.1.166+):** Deny rules in `settings.json` support glob patterns in the tool-name position — `"*"` denies all tools. Allow rules reject non-MCP globs. Unknown tool names in deny rules emit a startup warning.

**Per-parameter `Tool(param:value)` deny syntax (v2.1.178+):** Deny rules now support `Tool(param:value)` syntax to match a tool's input parameters, with `*` as a wildcard (e.g. `Agent(model:opus)` denies Opus subagent spawns). Extends v2.1.166 tool-name globs to per-parameter granularity. Complements `availableModels` ([[r006]]) for model-level access control and `mode: "bypassPermissions"` ([[r010]]) for subagent scope enforcement.

**Apple Events opt-in (v2.1.181+):** `sandbox.allowAppleEvents` setting added — opt-in to allow macOS Apple Events in the sandbox (default: off). Relevant when agents need to interact with GUI apps or AppleScript automation on macOS.

**Auth-stub MCP tools hidden in headless mode (v2.1.183+):** Tier-6 MCP auth-stub tools (those that exist solely to prompt the user to authenticate an MCP server) are no longer exposed to the model in headless/SDK (`-p`) mode. Previously, these stubs could surface in non-interactive runs and cause spurious tool-call attempts. No impact on interactive sessions.

**CLI MCP login + named-spawn deny enforcement (v2.1.186+):** Added `claude mcp login <name>` / `claude mcp logout <name>` to authenticate MCP servers from the CLI without the interactive `/mcp` menu (`--no-browser` stdin redirect for SSH). Also fixed `Agent(type)` deny rules and `Agent(x,y)` allowed-types restrictions not being enforced for named subagent spawns — extends the v2.1.178 `Tool(param:value)` per-parameter deny syntax to reliable enforcement on named spawns. Relevant to Tier-6 MCP and the Agent Tool Permission Mode.

**Org model restrictions on picker/env (v2.1.187+):** Org-configured model restrictions now apply to the model picker, `--model`, `/model`, and `ANTHROPIC_MODEL`, surfacing a "restricted by your organization's settings" message for a restricted model. Extends the v2.1.175 `enforceAvailableModels` managed-setting scope ([[r006]]) to the model picker/env entry points. Also added the `sandbox.credentials` setting (blocks sandboxed reads of credential files/secret env) — cross-ref [[r001]].

**Sandbox network memory + `/permissions` persist + MCP retry (v2.1.191+):** Sandbox network permission dialog now remembers hosts allowed with "Yes" for the rest of the session (no per-connection re-prompt). `/permissions` Recently-denied tab now persists an approved denial on close (previously discarded on close). Managed `forceRemoteSettingsRefresh` now takes effect via MDM/file policy with `Cache-Control: no-cache`. MCP capability discovery (`tools/list`/`prompts/list`/`resources/list`) and OAuth token requests now retry transient network errors with backoff (headless mode skips the browser popup). Relevant to Tier-4 (sandbox network) and Tier-6 (MCP) permission flows.

## Enforcement

Prompt-based per [[r021]]. On insufficient permission: do not attempt, notify user, suggest alternative.
