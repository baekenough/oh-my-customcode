# R016: Continuous Improvement Rules

When a user points out a rule violation, the relevant rule MUST be updated immediately — before continuing the original task.

## Overview

Rule violations reveal gaps or ambiguities in the rules themselves. Instead of just acknowledging the violation, the agent must update the rule to prevent recurrence, commit the change, and then resume the original task. Deferred fixes compound into patterns.

## Priority

**MUST** — Non-negotiable timing: rule update happens NOW, not later. "I'll fix it later" is itself a violation.

## Key Requirements

**Workflow:**
1. Acknowledge violation
2. Identify root cause (which rule was weak/unclear?)
3. Update the rule (add clarity, examples, self-checks)
4. Commit the change
5. Continue original task following updated rules

**Defect response matrix:**

| Defect Type | Rule Update | Memory | Issue |
|-------------|:-----------:|:------:|:-----:|
| Rule violation (agent behavior) | ✅ | — | — |
| CI/infra defect | — | ✅ | ✅ |
| Process gap (workflow hole) | ✅ | ✅ | ✅ |
| Repeatable system bug | — | ✅ | ✅ |

**Anti-patterns to avoid:**
- "I'll update the rule later"
- "This is a one-time exception"
- "Let me finish the task first"

## Enforcement

Prompt-based per [[r021]]. Rule updates require commit via [[r010]] → `mgr-gitnerd`.
