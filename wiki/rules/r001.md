# R001: Safety Rules

Defines absolutely prohibited actions and required verification steps before any destructive operation.

## Overview

Prevents agents from performing irreversible or dangerous actions such as exposing secrets, deleting system files, or making unauthorized network requests. Any potential violation triggers an immediate full stop and user notification.

## Priority

**MUST** — Non-negotiable. Safety violations can cause irreversible data loss or security breaches.

## Key Requirements

**Prohibited actions:**

| Category | Prohibited |
|----------|-----------|
| Data | Expose API keys/secrets/passwords, collect PII, log auth tokens |
| File System | Modify `/etc`, `/usr`, `/bin`; delete outside project; modify `.env`/`.git/config` without approval |
| Commands | `rm -rf /` or broad deletes, shutdown/restart, sudo/su, network config changes |
| External | Access URLs without approval, send user data externally, download/execute unknown scripts |

**Before any destructive operation:**
1. Verify target
2. Assess impact scope
3. Check recoverability
4. Get user approval

**On violation:**
1. Stop all operations
2. Preserve current state
3. Report: what detected, why risky, action taken
4. Wait for instructions

## Destructive Git Commands (v0.136.1, #1146)

Added in v0.136.1 after the v0.136.0 working tree loss incident. The following git commands require explicit user approval per invocation:

| Command | Risk | Required Action |
|---------|------|----------------|
| `git reset --hard <ref>` | Erases uncommitted + committed local changes | Confirm `git status` clean; show ref delta; explicit approval |
| `git checkout -- <path>` / `git restore <path>` | Discards uncommitted file changes | Confirm intentional revert; explicit approval |
| `git clean -fd` / `git clean -fdx` | Permanently deletes untracked files | List with `git clean -nd` first; explicit approval |
| `git branch -D <name>` (unmerged commits) | Loses unmerged work | Show `git log <branch>` first; confirm pushed; explicit approval |
| `git push --force` / `--force-with-lease` to shared branches | Rewrites shared history | NEVER on main/master; explicit approval for feature branches |

Recovery hint: `git reflog` — most operations recoverable within 30 days.

See [[git-safety]] for full pre-flight checks, recovery procedures, and agent-specific rules.

### Pre-Delegation Blast-Radius Enumeration (#1307)

Before delegating ANY destructive git command, enumerate the EXACT discard targets first — not just the symptom the user named. Run `git status --short` + `git diff --stat` and present the full scope for explicit approval. Prefer `git stash` when the user's goal can be met without permanent loss.

## Infra/Resource Deletion Blast-Radius (v0.177.0, #1327)

The git blast-radius enumeration principle generalizes to ALL infra/resource deletion (tunnels, DNS records, k8s resources, load balancers, security groups). Before deleting a shared infra resource, enumerate EVERY endpoint/hostname/route it serves — not only the one the user named.

| Resource | Enumerate before delete |
|----------|-------------------------|
| Tunnel (cloudflared, etc.) | All hostnames/DNS records routed through the tunnel — not just the named hostname |
| DNS record / zone | All services resolving via the record |
| k8s resource (Service, Ingress, etc.) | All selectors/endpoints/routes it backs |
| Load balancer / Security group | All targets/rules attached |

Present the full served-endpoint list for explicit approval before deletion. **Active-connection=0 on one hostname does NOT prove the resource is unused by others.** (Origin: #1327 찐빠 #3 — Cloudflare tunnel deleted after checking only the user-named hostname.)

## Credential & Privileged-Scope Guardrails

Prohibits dumping credential stores into transcripts, unrequested credential rotation, chaining approved actions into adjacent privileged operations, and irreversible shared-infra actions without scope re-confirmation.

| Prohibited | Required instead |
|-----------|------------------|
| Dumping `.env`, OAuth tokens, k8s secrets, `PG_DSN` into transcript/output | Reference secrets by name only; never echo values |
| Unrequested credential rotation / secret recreation | Rotate only on explicit user request, scoped to the specific secret |
| Chaining approved privileged action into unrequested adjacent ones | Each privileged op requires its own authorization trace |
| Irreversible shared-infra action without scope re-confirmation | Re-confirm scope with user before executing |

**Ask-before-scan (v0.177.0, #1327 찐빠 #4):** When a credential/token is needed, request it from the user BEFORE running system-wide credential scans (`env | grep`, workspace token greps). Such scans trip the Credential Exploration classifier. If a scan trips the classifier, stop and ask — do NOT retry the scan.

### Infra-Diagnostic File Checks — Metadata, Not Contents (v0.179.0, #1334)

When diagnosing infrastructure or health issues (502s, container state, missing config), file checks MUST use metadata-only commands — `ls -la` (existence, size, perms, mtime). Never use `cat .env`, `cat credentials.json`, or any command that reads secret contents or keys into the transcript, even to "confirm config is present". Confirming a file EXISTS is a metadata check; reading its values is a credential scan.

| Anti-pattern | Required |
|--------------|----------|
| `cat .env` / inspect OAuth key values to confirm config present | `ls -la .env` — existence/size/perms only; request a specific value from the user only if genuinely needed |

Cross-reference: Ask-before-scan (above), [[r010]] Subagent Scope-Creep STOP Protocol.

### Standing User-Deny + Classifier Block → Immediate user-runs Switch (v0.179.0, #1335)

When the user has a standing "don't touch X" constraint AND the safety classifier blocks an action on X even once, immediately switch to the `!` user-runs pattern — surface the exact command for the user to run themselves — and do NOT retry the blocked edit via a different mechanism.

| Anti-pattern | Required |
|--------------|----------|
| Retry a blocked edit on a user-deny-listed path via another mechanism | Stop after the first block; emit the exact command via `!` for the user to run; wait |

A standing deny + one classifier trip is a hard signal to delegate to the user. (Origin: #1335 찐빠 #4 — `.env.local` edit retried and blocked repeatedly despite a standing "절대 시크릿 건드리지 마" constraint.)

Cross-reference: [[r010]] Subagent Scope-Creep STOP Protocol (2-trip stop), [[r015]] Failed Tool Re-Try Discipline.

Cross-reference: [[r010]] Subagent Scope-Creep STOP Protocol, [[r002]] permission tiers.

**Sandboxed credential read blocking (v2.1.187+):** The `sandbox.credentials` setting blocks sandboxed commands from reading credential files and secret environment variables. Platform-level complement to the credential guardrails above — the model still never echoes secret values, and CC now also blocks sandboxed reads of credential files/secret env at the platform level. Defense-in-depth. Cross-ref [[r002]].

**Sandbox network permission session memory (v2.1.191+):** Sandbox network permission "Yes" approvals are remembered for the duration of the session (cf. [[r002]] permission tiers). This reduces repeated prompts for the same host, but also means an allowed host stays allowed for the rest of the session — scope network allows deliberately, as approval is not per-request once granted.

## Platform-Level Destructive Command Blocking (v2.1.183+)

Auto mode now platform-blocks (not merely advisory) a set of destructive commands when the user has not explicitly requested discarding:

| Command | Platform block condition |
|---------|--------------------------|
| `git reset --hard`, `git checkout -- .`, `git clean -fd`, `git stash drop` | Blocked unless user explicitly requested discarding changes in this request |
| `git commit --amend` | Blocked unless the commit being amended was made by the agent in the current session |
| `terraform/pulumi/cdk destroy` | Blocked unless a specific stack is named in the request |

**Defense-in-depth relationship:** R001's model-level advisory approval requirements (above) remain in effect. The v2.1.183 platform-level block is a *complementary* second layer — it gates the action at the platform before the model even attempts it, but does not replace the model-level pre-flight checks (target enumeration, blast-radius disclosure, user approval). Both layers apply together.

## Enforcement

Prompt-based (CLAUDE.md + rules/) per [[r021]]. Some specific patterns enforced by `stage-blocker` hard-block hook. Destructive git command rules are advisory at the model level; v2.1.183+ adds platform-level blocking for the most common destructive commands (see above).
