# R006: Agent Design Rules

Defines the file format, required frontmatter, optional fields, naming conventions, and separation-of-concerns architecture for all agents and skills.

## Overview

Agents live in `.claude/agents/{name}.md`. Skills live in `.claude/skills/{name}/SKILL.md`. Guides in `guides/`. These three locations have distinct purposes and must not be conflated. The agent frontmatter defines identity, tools, model, and behavior. The body describes role and workflow ŌĆö not detailed instructions (those belong in skills).

## Priority

**MUST** ŌĆö Agent files with invalid frontmatter or wrong structure cause routing failures and R017 verification failures.

## Key Requirements

**Required frontmatter fields:** `name`, `description`, `model`, `tools`

**Model aliases:**

| Alias | Full ID | Use Case |
|-------|---------|----------|
| `haiku` | claude-haiku-4-5 | Fast, cheap tasks |
| `sonnet` | claude-sonnet-4-6 | General tasks (default) |
| `opus` | claude-opus-4-6 | Complex reasoning |
| `opusplan` | claude-opus-4-6 + plan mode | Architecture planning |
| `fable` | claude-fable-5 | Mythos-class; **tier above Opus**, highest GA capability on the Claude API. CC v2.1.170 is the client version that adds access (model GA is a Claude API/platform property, not a CC-release milestone). Reserve for the most complex reasoning; sonnet remains default, opus for architecture (R005 cost-awareness). |

**Separation of concerns:**

| Location | Purpose | Contains |
|----------|---------|----------|
| `.claude/agents/` | WHAT the agent does | Role, capabilities, workflow |
| `.claude/skills/` | HOW to do tasks | Instructions, scripts, rules |
| `guides/` | Reference docs | Best practices, tutorials |

**Skill scope:**

| Scope | Deployed via init? |
|-------|--------------------|
| `core` | Yes |
| `harness` | Yes |
| `package` | No |

**Protected paths:** `.claude/agents/*.md`, `.claude/skills/*/SKILL.md`, `guides/*/` (new dirs) ŌĆö creation MUST go through `mgr-creator` per [[r010]].

## Artifact Output Convention

Skills persist output to `.claude/outputs/sessions/{YYYY-MM-DD}/{skill-name}-{HHmmss}.md`. Opt-in, git-untracked. Final subagent writes per [[r010]].

## Sensitive Path Handling

> **Status (CC v2.1.121+):** Direct Write/Edit/Bash on `.claude/`, `.git/`, `.vscode/` works without prompts under `mode: "bypassPermissions"`. The historical `/tmp/*.sh` bypass pattern is deprecated. See #1101.

Under `bypassPermissions`, all `.claude/**` paths (including `.claude/outputs/**`, `.claude/agents/**`, `.claude/skills/**`, `.claude/rules/**`, `templates/.claude/**`) accept Write/Edit/Bash directly. Catastrophic shell operations remain blocked by independent safety guards.

**Recommended practice:**
1. Pass `mode: "bypassPermissions"` on every Agent tool call (R010 Universal bypassPermissions)
2. Use Write/Edit directly for `.claude/**` paths ŌĆö no `/tmp/*.sh` wrapping needed
3. For CC < v2.1.121: the legacy `/tmp/*.sh` bypass pattern applied ŌĆö see git history

Note: `.claude/**` glob does not cover `templates/.claude/**` ŌĆö separate allow rules still required in non-bypassPermissions modes.

## CC Platform Settings (v2.1.166+ŌĆōv2.1.169+)

**Fallback models (v2.1.166+):** `fallbackModel` in settings configures up to three fallback models tried in order when the primary is overloaded or unavailable. Complements the per-agent `model:` field; handled at the platform level rather than per-agent frontmatter.

**Thinking toggle (v2.1.166+):** `MAX_THINKING_TOKENS=0`, `--thinking disabled`, or the per-model toggle disables thinking on models that think by default. Useful when an agent's `effort` is low and thinking overhead is undesirable.

**VS Code transcript bug (v2.1.170 fix):** Sessions started from the VS Code integrated terminal no longer fail to save the session transcript (and transcripts now appear in `--resume`). Skills that depend on episodic memory or transcript replay (e.g., `homework`, `episodic-memory`) are affected when this bug is present ŌĆö upgrade to CC v2.1.170+ to ensure transcripts are saved in VS Code terminal sessions.

**Safe Mode (v2.1.169+):** `--safe-mode` (env: `CLAUDE_CODE_SAFE_MODE`) starts CC with ALL customizations disabled ŌĆö CLAUDE.md, plugins, skills, hooks, MCP servers. Use it to isolate whether a project agent/skill/hook causes a regression.

**Bundled skill control (v2.1.169+):** `disableBundledSkills` setting (env: `CLAUDE_CODE_DISABLE_BUNDLED_SKILLS`) hides CC's bundled skills, workflows, and built-in slash commands from the model. Useful when bundled skills conflict with or duplicate project skills. Distinct from the advisory `skills:` frontmatter field, which is documentation metadata and not a runtime allowlist ŌĆö see the `skills:` note in the source rule.

**`availableModels` subagent overrides (v2.1.172+):** `availableModels` restrictions now apply to subagent `model:` overrides, the agent dispatch model picker, and the advisor model. Allowlists using version-specific IDs (e.g. `claude-opus-4-8`) no longer hide the Opus/Sonnet 1M picker rows, and model IDs no longer receive a doubled 1M suffix (`[1M][1m]`) when `ANTHROPIC_DEFAULT_OPUS_MODEL` already includes one. Relevant when restricting per-agent model overrides via `availableModels`.

**Fable 5 `[1m]` auto-normalization (v2.1.173+):** Fable 5 model IDs carrying a `[1m]` suffix are auto-normalized (suffix stripped) because Fable 5 includes 1M context by default. Use `claude-fable-5` / `model: fable` WITHOUT a `[1m]` suffix ŌĆö appending it is redundant. The `[1m]` suffix remains meaningful for Opus/Sonnet IDs.

**`enforceAvailableModels` managed setting (v2.1.175+):** When enabled, the `availableModels` allowlist also constrains the **Default** model ŌĆö a Default that would resolve to a disallowed model falls back to the first allowed model, and user or project settings can no longer widen a managed `availableModels` list. Extends the v2.1.172 `availableModels` scope (subagent `model:` overrides, dispatch picker, advisor model) to the Default model itself. Relevant in enterprise/managed configs that pin the allowed model set: both per-agent `model:` overrides AND the resolved Default honor the allowlist.

**CC v2.1.178 updates:**

- **Compaction + fallback chain (v2.1.178+):** Compaction now honors the `fallbackModel` chain on overload/availability errors that occur during compaction. Previously, compaction could fail when the primary model was unavailable and the fallback chain was not consulted; this ensures compaction remains resilient when using `fallbackModel` settings.

- **Nested `.claude/skills` loading (v2.1.178+):** Nested `.claude/skills` directories now load with `<dir>:<name>` disambiguation on name clash (avoids silent overrides when multiple nested directories define a skill of the same name). Additionally, MCP-spec entries (`mcp__*`) in a subagent's `disallowedTools` frontmatter are now honored ŌĆö previously these were silently ignored.

- **Nearest `.claude/` wins (v2.1.178+):** On agent/workflow/output-style name collisions across nested `.claude/` directories, the nearest (innermost) `.claude/` directory takes precedence. Relevant when working in a nested repo or monorepo layout where multiple `.claude/` directories coexist.

**CC v2.1.183 updates:**

- **Deprecated model warning covers frontmatter (v2.1.183+):** When the requested model is deprecated or auto-updated by the platform, CC now emits a warning to stderr (and in `-p` headless mode). This warning now also covers the agent frontmatter `model:` field ŌĆö agents with a deprecated `model:` value will surface the warning when the agent is spawned. Update deprecated model aliases in agent frontmatter proactively when the warning appears.

- **Thinking-disabled 400 fix (v2.1.183+):** Fixes `thinking.disabled.display: Extra inputs are not permitted` 400 errors that could occur during subagent spawn or session-title generation on models where thinking is disabled. Extends the v2.1.166 Thinking Toggle ŌĆö agents using `MAX_THINKING_TOKENS=0` / `--thinking disabled` no longer hit this 400 on spawn or title generation.

**CC v2.1.187 updates:**

- **Org model restrictions cover picker/env (v2.1.187+):** Org-configured model restrictions now apply to the model picker, `--model`, `/model`, and `ANTHROPIC_MODEL` (a restricted model shows "restricted by your organization's settings"). Extends the v2.1.175 `enforceAvailableModels` scope to the per-agent `model:` override entry points ŌĆö a managed model allowlist now also constrains the picker/env paths.

- **Structured-output re-call fix (v2.1.187+):** Fixed `--json-schema` / workflow `agent({schema})` structured output ŌĆö the model can no longer re-call `StructuredOutput` indefinitely after a successful call, and follow-up turns reliably return structured output. Relevant to schema-constrained subagent spawns.
